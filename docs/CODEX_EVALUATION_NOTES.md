# Codex — зафиксированные замечания по ходу эксперимента

## Назначение документа

Этот файл фиксирует фактические замечания, возникшие при практической работе с Codex над `my-tracker-codex`.

Документ не является окончательным выводом о качестве Codex как продукта. Он описывает конкретный пользовательский опыт в рамках этого эксперимента и должен учитываться вместе с качеством итоговой реализации.

## Контекст

- Codex первоначально запускался в локальном режиме на Windows-машине пользователя.
- Репозиторий: `Wdivin/my-tracker-codex`.
- Локальное окружение пришлось подготавливать через OpenServerPanel.
- Для Cloud-режима подключение GitHub потребовало обязательного включения MFA в аккаунте OpenAI.
- Пользователь не хочет использовать MFA из-за неудобства, поэтому Cloud-режим фактически оказался недоступен для этого сценария.

## Зафиксированные претензии

### 1. Зависимость от локального окружения

В локальном режиме Codex зависит от программного обеспечения и сервисов, установленных на пользовательской машине.

Для продолжения работы потребовалось вручную подготовить:

- Apache;
- PHP 8.4;
- MySQL 8.4;
- Redis;
- OpenServerPanel и его конфигурацию.

До подготовки окружения Codex сообщил, что не может корректно завершить этап, поскольку отсутствуют Docker и доступная MySQL.

Это увеличило объём ручной работы пользователя и время запуска задачи.

### 2. Неравные условия по сравнению с Claude

Claude выполнял работу в собственном подготовленном окружении и самостоятельно обеспечивал наличие PHP, MySQL, Redis, Node.js и браузерных инструментов.

Codex в локальном режиме потребовал настройки пользовательской машины.

Это делает исходные условия эксперимента неравными и создаёт дополнительную стоимость сопровождения именно для Codex.

### 3. Частые остановки после промежуточных действий

Codex неоднократно завершал текущий ход после:

- создания commit;
- создания Draft PR;
- запуска GitHub Actions;
- ожидания результата CI;
- исправления отдельной ошибки CI;
- завершения части внутреннего этапа.

При этом общая задача заключалась в самостоятельной реализации полной версии 1.0 без промежуточного согласования.

Пользователю приходилось повторно отправлять команды вида «продолжай работу».

### 4. Остановка на ожидании CI

Codex останавливался после запуска GitHub Actions и сообщал, что CI находится в ожидании или ещё не зарегистрировал checks.

Он не продолжал автоматически после завершения внешнего процесса и требовал нового пользовательского хода.

Для длинной задачи это создаёт необходимость регулярно следить за окном агента.

### 5. Недостаточная автономность длинной задачи

Даже после явной команды:

- не останавливаться после внутренних этапов;
- самостоятельно исправлять CI;
- переходить к следующим этапам без утверждения;
- обращаться только при настоящем внешнем блокере;

Codex продолжал завершать ходы после промежуточных технических результатов.

В рамках цели «дать большую задачу и получить готовый результат» это оценивается как недостаток автономности.

### 6. Необходимость ручного контроля состояния

Пользователь не может постоянно наблюдать за окном Codex и определять, остановился ли он.

В ходе эксперимента не было получено удобного и очевидного уведомления о том, что:

- агент закончил текущий ход;
- агент ждёт нового сообщения;
- агент остановился после ожидания CI;
- требуется действие пользователя.

Отсутствие понятного уведомления повышает операционные затраты.

### 7. Cloud-режим требует MFA

Для подключения GitHub к Codex Cloud интерфейс потребовал включить многофакторную аутентификацию OpenAI.

Пользователю неудобно работать с MFA, а поддерживаемого способа подключить GitHub к Codex Cloud без MFA не обнаружено.

Следствие:

- автономный Cloud-режим формально существует;
- но для данного пользователя и сценария он практически недоступен без изменения предпочтений безопасности аккаунта;
- остаётся локальный режим со всеми перечисленными ограничениями.

### 8. Повышенная стоимость пользовательского сопровождения

В ходе работы с Codex пользователю пришлось:

- поднимать и настраивать локальные сервисы;
- повторно давать команды на продолжение;
- следить за завершением CI;
- выяснять причины отсутствия checks;
- отдельно исследовать возможность Cloud-режима;
- разбираться с обязательным MFA.

Эти действия должны учитываться при финальной оценке не только как удобство интерфейса, но и как реальная стоимость сопровождения разработки.

## Что не следует считать претензией

Следующие факты сами по себе не являются недостатками:

- использование GitHub Actions для чистой независимой проверки;
- создание промежуточных commits и PR;
- разбиение внутреннего плана на этапы;
- необходимость исправлять реальные ошибки CI.

Претензия относится не к наличию этих процессов, а к тому, что Codex неоднократно прекращал работу после них и требовал ручного запуска следующего хода.

## Как учитывать при финальной оценке

Эти наблюдения следует учитывать в критериях:

- автономность исполнения;
- количество обязательных вмешательств пользователя;
- удобство фоновой работы;
- воспроизводимость рабочего окружения;
- стоимость запуска и сопровождения;
- способность самостоятельно довести большую задачу до конечного результата.

Для честной оценки нужно отдельно подсчитать:

- число остановок без настоящего внешнего блокера;
- число повторных команд «продолжай»;
- число ручных действий по настройке среды;
- время пользователя, потраченное на сопровождение;
- качество и полноту финального продукта.

Окончательный вывод должен учитывать и автономность процесса, и техническое качество результата. Слабая автономность не должна скрываться высоким качеством кода, но и не должна автоматически обесценивать сильный итоговый продукт.
