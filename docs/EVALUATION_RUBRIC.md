# My Tracker — матрица оценки реализации

## 1. Назначение

Матрица используется для приёмки реализации `IMPLEMENTATION_BRIEF v1.0` и последующей объективной оценки качества результата по заранее определённым правилам.

Оценка не заменяет обязательные критерии приёмки brief. Реализация с критическим нарушением безопасности, потерей данных, неработающим развёртыванием или отсутствующим ключевым рабочим циклом не может быть принята только за счёт высокого среднего балла.

Максимальная итоговая оценка — **100 баллов**.

## 2. Блокирующие условия

До начисления баллов проверяются блокирующие условия. Реализация не допускается к итоговой приёмке, если присутствует хотя бы одно из следующего:

- приложение нельзя воспроизводимо установить и запустить по документации;
- отсутствует рабочая аутентификация и server-side проверка прав;
- отсутствует основной цикл `создать задачу → назначить → изменить → провести по workflow → закрыть`;
- Web UI и API применяют разные бизнес-правила или позволяют обходить права;
- обнаружена воспроизводимая потеря или неконтролируемое повреждение данных;
- миграции не работают на чистой БД;
- критические секреты попадают в репозиторий, логи, аудит или клиентский код;
- присутствует известная critical vulnerability без устранения;
- отсутствует фактически работающая мобильная версия ключевых сценариев;
- отсутствуют обязательные инструкции запуска и проверки.

Если блокирующее условие можно устранить ограниченной правкой, результат возвращается на доработку. Если проблема носит архитектурный характер, это учитывается при итоговом решении даже после исправления.

## 3. Шкала оценки критерия

Для каждого критерия используется единая шкала:

- **0% веса** — функция отсутствует, не работает или решение неприемлемо;
- **25% веса** — реализован фрагмент, есть серьёзные дефекты или обход требований;
- **50% веса** — основной сценарий работает, но заметно неполон или нестабилен;
- **75% веса** — качественная реализация с ограниченными недостатками;
- **100% веса** — полная, надёжная, понятная и хорошо проверенная реализация.

Допускаются промежуточные оценки, если они обоснованы конкретными результатами проверки.

## 4. Критерии и веса

### 4.1. Полнота продуктового объёма — 18 баллов

Проверяется:

- покрытие обязательных разделов `IMPLEMENTATION_BRIEF.md`;
- отсутствие молча исключённых функций;
- завершённость пользовательских сценариев, а не наличие пустых экранов;
- корректная реализация отложенных функций как именно отложенных, без недостоверных заглушек;
- наличие рабочего ядра и последовательное доведение до 1.0.

Полный балл требует реализации всего обязательного объёма и понятного списка действительно известных ограничений.

### 4.2. Бизнес-правила и целостность данных — 14 баллов

Проверяется:

- задачи, подзадачи, связи и защита от циклов;
- проекты, подпроекты, глубина и наследование;
- workflow, переходы, обязательные поля и закрывающие статусы;
- максимум один исполнитель;
- перенос задачи и смена типа без скрытой потери данных;
- мягкое удаление, восстановление и физическое удаление;
- транзакционность составных операций;
- optimistic locking в обязательных областях;
- идемпотентность обязательных API-операций.

Особо штрафуются молчаливое затирание данных и возможность обхода доменных правил.

### 4.3. Права, безопасность и аудит — 12 баллов

Проверяется:

- server-side авторизация каждого действия;
- системные и проектные роли;
- наследование участников и ролей;
- отсутствие утечек закрытых проектов и задач;
- безопасные токены, rate limit и request ID;
- безопасность файлов;
- отсутствие секретов в истории и аудите;
- корректность системного аудита;
- безопасные административные действия;
- обработка деактивации пользователя;
- dependency/security checks.

### 4.4. Desktop UX и визуальное качество — 9 баллов

Проверяется:

- понятность без отдельного обучения;
- последовательная терминология;
- качество списков, карточки задачи, доски, фильтров и администрирования;
- progressive disclosure;
- состояния загрузки, ошибки, пустые состояния и подтверждения;
- отсутствие наложений, обрезаний и ненужной горизонтальной прокрутки;
- удобство inline-редактирования и опасных операций;
- визуальная целостность, а не набор стандартных CRUD-форм.

### 4.5. Mobile UX и PWA — 9 баллов

Проверяется:

- полноценность ключевых мобильных сценариев;
- удобная навигация пальцем;
- карточное представление вместо сжатых таблиц;
- создание и редактирование задачи;
- workflow, комментарии, файлы, фильтры и время;
- работа на согласованных узких разрешениях;
- устанавливаемая PWA;
- отсутствие заявлений об offline-функциях, которые не входят в 1.0.

### 4.6. REST API и OpenAPI — 8 баллов

Проверяется:

- версионированный JSON REST API;
- соответствие Web и API одной доменной логике;
- OpenAPI, реально соответствующий реализации;
- пагинация, фильтры и сортировка;
- единый формат ошибок;
- scopes и права service users;
- idempotency;
- optimistic locking;
- безопасная загрузка файлов;
- partial-success bulk operations;
- пригодность API для практической интеграции.

### 4.7. Архитектура и сопровождаемость — 8 баллов

Проверяется:

- ясность модульных границ;
- отсутствие неоправданной сложности;
- качество доменной модели;
- разделение бизнес-логики и транспорта;
- качество миграций и индексов;
- расширяемость без преждевременной универсализации;
- понятность кода и структуры проекта;
- документирование ключевых решений и компромиссов;
- пригодность к дальнейшей разработке небольшими задачами.

Оценивается не конкретный выбранный стек, а качество его применения.

### 4.8. Автоматические тесты и проверяемость — 8 баллов

Проверяется:

- unit/integration/feature tests критичной логики;
- тесты прав и workflow;
- тесты конкурентных изменений;
- тесты API и файлов;
- E2E ключевых сценариев;
- воспроизводимые demo/seed data;
- отсутствие фиктивных тестов, проверяющих только код ответа без бизнес-результата;
- стабильность и понятность команд запуска тестов.

### 4.9. Производительность и масштабируемость — 5 баллов

Проверяется:

- серверная пагинация;
- индексы основных выборок;
- отсутствие очевидных N+1 и загрузки всей истории;
- разумная работа списков, поиска и фильтров на представительном наборе данных;
- фоновые очереди для почты и тяжёлых действий;
- отсутствие архитектурных ограничений на несколько сотен задач;
- документированная методика измерений и ограничения среды.

Миллион задач является архитектурным ориентиром, а не обязательной нагрузочной сертификацией.

### 4.10. Развёртывание, эксплуатация и backup — 5 баллов

Проверяется:

- clean install;
- upgrade существующей установки;
- production build;
- очереди и scheduler;
- SMTP и тестовая отправка;
- файловое хранилище;
- health/status диагностика;
- документированный deployment;
- реально выполненная проверка backup/restore;
- отсутствие ручного редактирования БД как штатного шага.

### 4.11. Документация и итоговый отчёт — 4 балла

Проверяется:

- README и установка;
- production deployment и upgrade;
- backup/restore;
- API/OpenAPI;
- руководство администратора и краткое руководство пользователя;
- troubleshooting;
- архитектурные решения;
- known limitations;
- полнота финального отчёта по brief;
- соответствие документации фактическому поведению.

## 5. Штрафы

Дополнительно к баллам применяются прозрачные штрафы:

- до **−10** за обязательную функцию, заявленную реализованной, но фактически отсутствующую;
- до **−10** за существенный обход прав или workflow, не достигающий блокирующего уровня;
- до **−8** за систематические мобильные дефекты;
- до **−8** за несоответствие OpenAPI фактическому API;
- до **−6** за невоспроизводимый или нестабильный тестовый набор;
- до **−6** за недокументированное существенное ограничение;
- до **−5** за серьёзные визуальные дефекты, оставленные после обязательной браузерной проверки;
- до **−5** за необоснованное архитектурное переусложнение, существенно мешающее сопровождению.

Один дефект не должен механически штрафоваться несколько раз. Проверяющий указывает, к какому критерию и штрафу он отнесён.

## 6. Процедура проверки

Проверка выполняется в одинаковой последовательности:

1. зафиксировать commit и документацию результата;
2. выполнить clean install по инструкции;
3. запустить миграции и seed/demo data;
4. выполнить backend/frontend tests, build, lint и security checks;
5. пройти обязательные E2E-сценарии;
6. проверить desktop и mobile разрешения;
7. проверить права минимум трёх проектных ролей и системного администратора;
8. проверить Web/API согласованность;
9. проверить migration upgrade;
10. провести backup/restore drill;
11. зафиксировать дефекты и доказательства;
12. начислить баллы и штрафы с кратким обоснованием.

Проверяющий не изменяет реализацию до завершения первичной оценки. Исправления после первичной оценки учитываются отдельно, чтобы было видно качество исходного результата и качество реакции на замечания.

## 7. Итоговая оценка

Итог содержит:

- статус блокирующих условий;
- баллы по каждому разделу;
- применённые штрафы;
- общий результат из 100;
- список critical/high/medium/low дефектов;
- сильные стороны;
- архитектурные риски;
- ориентировочную стоимость дальнейшего сопровождения;
- рекомендацию: принять, принять после исправлений или отклонить.

При близких итоговых баллах приоритет получают:

1. корректность и безопасность;
2. полнота бизнес-правил;
3. качество mobile/desktop UX;
4. сопровождаемость;
5. тесты и воспроизводимость;
6. производительность.

Более эффектный внешний вид не компенсирует потерю данных, обход прав, неполный workflow или невоспроизводимое развёртывание.
