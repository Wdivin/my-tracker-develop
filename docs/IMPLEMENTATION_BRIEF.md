# My Tracker — IMPLEMENTATION BRIEF v1.0

## 1. Статус и назначение

Версия документа: **1.0**.

Статус: **утверждённое единое задание для независимой реализации**.

Документ предназначен для передачи двум независимым исполнителям — Codex и Claude Code. Каждый исполнитель самостоятельно проектирует и реализует продукт, не используя код, ветку или решения второго исполнителя.

Этот файл является главным источником требований для реализации версии 1.0. При конфликте действует приоритет:

1. этот `IMPLEMENTATION_BRIEF.md`;
2. более позднее утверждённое решение из `docs/decisions`;
3. более раннее решение из `docs/decisions`;
4. `docs/PRODUCT_SPEC.md`;
5. README и прочие пояснительные документы.

Исполнитель не должен молча исключать обязательную функцию. Неясности разрешаются в пользу целостности данных, безопасности, простоты для пользователя и одинаковых правил Web/API. Существенные предположения и компромиссы нужно перечислить в отчёте.

## 2. Цель продукта

Создать современный универсальный корпоративный трекер задач, использующий сильные предметные идеи Redmine/Jira, но не копирующий их интерфейс и исторические технические решения.

Главный принцип:

> **Гибкость корпоративной системы внутри, простота современного приложения снаружи.**

Продукт должен подходить для IT, поддержки, заявок, согласований, маркетинга, административных и других внутренних процессов.

Ключевые принципы:

- сложность раскрывается постепенно;
- мобильный интерфейс — полноценное рабочее место;
- у задачи максимум один основной исполнитель;
- скрытая автоматизация и неочевидное наследование запрещены;
- Web UI и REST API используют одну бизнес-логику;
- история, архивирование и мягкое удаление предпочтительнее потери данных;
- необратимые и массовые действия безопасны по умолчанию;
- интерфейс не должен выглядеть как набор сырых CRUD-форм.

## 3. Объём и этапность

Полный объём этого документа является **целевой версией 1.0**.

Реализацию нужно разбить на последовательные этапы. Первый этап — **рабочее ядро**, пригодное для реального использования. Затем продукт последовательно доводится до полного объёма 1.0.

До начала массовой реализации исполнитель должен:

1. изучить весь brief;
2. предложить архитектуру и этапы;
3. описать зависимости и риски;
4. обеспечить возможность запуска и проверки после каждого этапа.

Нельзя трактовать версию 1.0 как требование сделать всё одним огромным изменением.

## 5. Организация и пользователи

Одна установка обслуживает одну организацию. Multi-tenancy не входит в 1.0, но архитектура не должна намеренно делать его невозможным.

Организация содержит название, логотип, основные локальные настройки, часовой пояс, форматы дат, настройки уведомлений, файловые лимиты и параметры безопасности.

Пользователи:

- создаются системным администратором;
- могут приглашаться по ссылке для установки пароля;
- имеют локальную учётную запись;
- могут восстановить пароль;
- выбирают язык и часовой пояс;
- деактивируются без удаления истории;
- не регистрируются публично самостоятельно.

Деактивация завершает активные сессии, отзывает токены и исключает пользователя из новых назначений. Старые задачи, комментарии, время и история сохраняются. Автоматического переназначения задач нет.

Интеграция Active Directory/LDAP, SAML и OAuth-вход **не входят в версию 1.0**.

## 6. Системные администраторы

Системный администратор может:

- создавать и деактивировать пользователей;
- назначать другим пользователям права системного администратора;
- снимать права системного администратора у других;
- управлять глобальными справочниками, типами задач, workflow, полями, ролями, настройками, токенами, корзиной и аудитом.

Администратор:

- не редактирует и не подменяет аудит;
- не видит пароли и полные секреты;
- не получает универсальный обход workflow обычных задач;
- выполняет критические действия с подтверждением и аудитом.

Начальная установка создаёт первый локальный административный аккаунт.

## 7. Проекты и подпроекты

Проект — основной рабочий контейнер. Подпроект — тот же проект с родителем.

Разрешено максимум два уровня подпроектов:

```text
Корневой проект
└── Подпроект первого уровня
    └── Подпроект второго уровня
```

Циклы запрещены. Перенос проекта должен проверять глубину и циклические связи.

Основные данные проекта:

- название;
- неизменяемый после создания уникальный код;
- автоматически созданный уникальный slug, доступный для изменения по праву;
- стабильный UUID/ULID;
- описание;
- родитель;
- владелец и руководитель;
- даты начала, планового и фактического завершения;
- статус;
- видимость;
- цвет или иконка;
- порядок;
- автор и даты создания/изменения.

Код используется в ключах задач, например `WEB-142`. UUID/ULID используется для внутренних связей и интеграций. Изменение slug не меняет стабильный ID; сложная история старых slug не обязательна.

Статусы проекта:

- `планируется` — работа разрешена по правам;
- `активный` — обычный режим;
- `приостановлен` — проект виден участникам, рабочие изменения по умолчанию запрещены;
- `завершён` — доступен участникам для просмотра, новые задачи и обычные изменения запрещены;
- `архивный` — скрыт для всех, кроме системных администраторов.

Возврат завершённого/архивного проекта требует отдельного права, подтверждения и аудита. Нельзя архивировать родительский проект при активных подпроектах.

Видимость:

- закрытый — видят участники и системные администраторы;
- внутренний — видят все активные пользователи организации, но без роли имеют только чтение.

Публичных проектов нет. Если пользователь видит проект, он видит все его задачи. Приватных задач и конфиденциальных полей в 1.0 нет.

## 8. Наследование и конфигурация проекта

Наследование конфигурации и наследование участников — независимые режимы.

Подпроект либо:

- полностью наследует конфигурацию родителя;
- либо использует собственную конфигурацию.

Частичное смешанное наследование запрещено.

Конфигурация включает:

- доступные типы задач;
- workflow для каждого типа;
- доступные проектные роли;
- проектные представления;
- глобальный или собственный полный набор видов работ.

Пользовательские поля проект напрямую не включает: набор полей определяется типом задачи.

Участники подпроекта могут:

- наследоваться с ролями родителя и дополняться локальными;
- либо использовать полностью собственный состав.

Интерфейс явно показывает источник участника/роли. Унаследованную роль нельзя локально отнять, пока наследование включено.

## 9. Роли, права и группы

Контексты доступа:

1. системное администрирование;
2. проектные роли;
3. отношение пользователя к задаче — автор, исполнитель, наблюдатель.

Проектные роли настраиваемые. Стартовые шаблоны:

- руководитель проекта;
- участник/исполнитель;
- читатель.

Пользователь может иметь несколько ролей; разрешения складываются. Явных deny-прав в 1.0 нет.

Права группируются по понятным областям: проект, задачи, поля, ответственность, структура, комментарии/файлы/чек-лист, время, участники, отчёты, массовые операции, удаление и восстановление.

Группы используются **только для прав**:

- группу можно добавить в проект и назначить ей роль;
- группа не может быть автором, исполнителем, наблюдателем, получателем уведомления или владельцем времени.

Все рабочие действия и уведомления привязаны к конкретным пользователям.

## 10. Задача

Основные данные:

- стабильный ID;
- человекочитаемый ключ из кода проекта и номера;
- проект;
- название и описание;
- тип;
- статус;
- приоритет;
- автор;
- максимум один исполнитель;
- дата создания/изменения;
- дата начала;
- срок;
- дата завершения;
- оценка времени;
- прогресс;
- теги;
- родительская задача;
- подзадачи;
- связи;
- чек-лист;
- вложения;
- добровольные наблюдатели;
- пользовательские поля.

Автор неизменяем. Исполнитель может отсутствовать.

Создание доступно глобально, из проекта, доски, как подзадача и через API. Минимум: проект, тип и название; дополнительные обязательные поля определяются типом и переходами.

Локальное автосохранение незавершённой формы не требуется для первого рабочего релиза.

## 11. Inline-редактирование и структурные операции

Inline допускается для обычных рабочих полей:

- название;
- описание;
- исполнитель;
- приоритет;
- дата начала;
- срок;
- прогресс;
- оценка;
- теги;
- простые пользовательские поля;
- наблюдение;
- чек-лист в собственном блоке.

Отдельным действием, переходом или мастером выполняются:

- смена статуса;
- перенос в другой проект;
- смена типа;
- смена родителя;
- удаление/восстановление;
- структурные операции с последствиями.

Перенос и смена типа должны заранее показывать несовместимые workflow, статусы, поля и значения. Молчаливое обнуление запрещено. Значения полей, скрывшихся после смены типа, сохраняются.

## 12. Подзадачи и связи

У задачи максимум один родитель и разумное количество подзадач. Циклы запрещены.

Связи минимум:

- связана с;
- блокирует / заблокирована;
- дублирует / дублируется.

Нельзя закрыть родительскую задачу при наличии незакрытых подзадач. Это жёсткое правило без обхода администратором.

Блокирующая связь не запрещает закрытие, но вызывает заметное предупреждение и подтверждение.

Нельзя мягко удалить задачу, пока у неё есть неудалённые подзадачи. Сначала их нужно удалить, перенести или отвязать.

## 13. Типы задач, поля и формы

Типы задач глобальные. Проект выбирает доступные типы.

Тип определяет:

- имя и иконку;
- включённые стандартные поля;
- пользовательские поля;
- обязательность стандартных и пользовательских полей;
- использование прогресса, времени и подзадач;
- порядок и секции полей.

Обязательные для целостности поля — проект, тип, название, статус, автор и ID — отключить нельзя.

Глобальные пользовательские поля назначаются выбранным типам или всем типам. Проектных пользовательских полей нет.

Типы пользовательских полей 1.0:

- короткий и многострочный текст;
- целое и десятичное число;
- дата и дата-время;
- boolean;
- одиночный и множественный выбор;
- ссылка на проект.

Не входят: пользователь, задача, файл, формулы, JSON, таблицы, геолокация, подпись, внешние динамические справочники.

Для поля настраиваются имя, административное описание, пользовательская подсказка, обязательность, порядок, секция, видимость, quick edit и простые ограничения. Обязательность возможна при создании, сохранении или конкретном переходе.

Разрешены только явные видимые default-значения. Скрытые поведенческие и AI-defaults запрещены.

Использованный тип поля нельзя произвольно менять. Использованные поля и значения справочников архивируются, а не удаляются.

Сложной условной видимости и зависимостей полей нет.

## 14. Workflow

Глобально существуют шаблоны workflow. Для пары `проект + тип задачи` проект выбирает глобальный шаблон либо независимую локальную копию.

Workflow содержит статусы и явные переходы. У него ровно один начальный статус. Начальный статус не дублируется в типе задачи.

Статус имеет явный признак закрывающего. Название не определяет закрытость. Категории статусов не требуются.

Пользователь меняет статус только через доступный переход. Универсального «поставить любой статус» нет, включая администратора.

Переход настраивает:

- доступные роли и отношения к задаче;
- обязательность комментария;
- обязательные поля;
- поддерживаемые продуктом проверки;
- уведомления и системные последствия.

Между одной парой статусов — максимум один переход. Переход может иметь пользовательское название действия.

Отдельных черновиков workflow нет. Сохраняется только целостная конфигурация. Недостижимые статусы — ошибка. Незакрывающий статус без исходящих переходов — предупреждение, которое администратор может подтвердить.

Обязательный комментарий перехода создаётся обычным комментарием, связан с событием и может редактироваться автором с сохранением редакций. Редактирование не отменяет переход.

## 15. Прогресс

Прогресс — ручной процент 0–100, независимый от статуса. Тип задачи может отключить поле.

Основной UX — ползунок:

- default-шаг 5%;
- системный выбор 1%, 5% или 10%;
- допускается точный числовой ввод;
- доступность мышью, клавиатурой и касанием.

При закрытии система предлагает установить 100%, но пользователь может отказаться. Закрытая задача может иметь 50% или другое значение. Переход может явно потребовать определённый прогресс.

Автоматический прогресс по статусу не используется.

## 16. Комментарии, Markdown и упоминания

Комментарии поддерживают безопасный Markdown, упоминания и вложения. Вложенных веток нет. Ответ оформляется обычным комментарием со ссылкой/цитатой и упоминанием.

Упомянуть можно только пользователя с доступом к задаче. Групповых упоминаний нет.

Автор может редактировать комментарий в любое время. Все редакции сохраняются, отображается отметка «изменено», старые версии открываются отдельно.

Автор может мягко удалить и восстановить свой комментарий. Отдельное право позволяет управлять чужими. На месте удалённого текста остаётся отметка. Комментарии физически отдельно не удаляются — только вместе с окончательным удалением задачи.

Просмотр содержимого и редакций удалённых комментариев регулируется отдельным проектным правом, включённым по умолчанию руководителю проекта; системный администратор имеет глобальный доступ.

## 17. История задачи и системный аудит

История задачи и системный аудит — разные механизмы.

История задачи фиксирует значимые изменения: поля, тип/проект, workflow, ответственность, сроки, прогресс, теги, поля, структуру, чек-лист, файлы, время, удаление и восстановление.

Одно сохранение нескольких полей создаёт сгруппированное событие. Переход, обязательный комментарий и связанные изменения отображаются связанно. Технические просмотры и UI-настройки в историю не попадают.

Лента объединяет комментарии, изменения, workflow, файлы, время и системные события; имеет серверную пагинацию/порционную загрузку и фильтры.

История хранится весь срок жизни задачи и удаляется после её окончательного физического удаления.

Хранение значений:

- короткие поля — old/new;
- длинный текст — версии или отдельное сравнение;
- коллекции — добавленные/удалённые элементы;
- файлы — метаданные и факт операции;
- секреты — только факт изменения без значений.

Системный аудит фиксирует входы, права, участников, workflow, справочники, токены, удаление/восстановление, настройки, массовые операции, экспорт и административные действия.

Аудит содержит применимые user, time, action, object, result, source, IP, ограниченный user agent и идентификатор интеграции/токена без секрета.

Retention по умолчанию — 365 дней, единый для 1.0; автоочистку можно отключить. Записи нельзя редактировать или выборочно удалять через UI. Очистка по retention создаёт агрегированное событие. WORM/криптографический журнал не требуется.

## 18. Файлы и вложения

Вложения поддерживаются у задач и комментариев. Inline-изображения являются защищёнными вложениями.

Desktop: file picker, multiple upload, drag-and-drop, clipboard images. Mobile: файл, фото, камера по возможностям браузера. Drag-and-drop не единственный путь.

Default-лимиты:

- 25 МБ на файл;
- 10 файлов за одну операцию;
- 500 МБ активных вложений на задачу.

Лимиты настраиваются. Квота считает только активные вложения.

Имена могут повторяться; физические ID не зависят от имени. Для изображений — preview/zoom/download, для PDF — безопасный browser viewer. Office preview через внешний сервис отсутствует.

Версий файлов нет. В старой редакции комментария сохраняется только факт и метаданные прежнего вложения, но не сам файл. Замена означает новый текущий файл без файлового versioning.

Вложения комментария изменяются только вместе с комментарием. Вложения задачи можно мягко удалять/восстанавливать по правам.

Безопасность:

1. системно опасные форматы блокируются всегда;
2. denylist приоритетнее allowlist;
3. заполненный allowlist разрешает только перечисленное;
4. пустой allowlist разрешает всё, кроме системного запрета и denylist;
5. проверяются расширение, MIME и сигнатура, когда возможно.

HTML и опасный SVG не показываются inline. Файлы хранятся вне публично исполняемой директории. Постоянных публичных ссылок нет; каждое скачивание проверяет текущий доступ.

ZIP хранится и скачивается, но не распаковывается, не исполняется, не preview-ится и не индексируется.

Незавершённые загрузки не являются вложениями, не входят в квоту и очищаются по TTL. Повторное завершение не создаёт дубль.

Антивирус архитектурно возможен, но не обязателен. Контроль orphan-файлов и integrity CLI — после 1.0.

## 19. Учёт времени

Только ручные записи, без таймера и автоматического tracking.

Запись содержит задачу, дату работы, длительность, обязательный вид работ, необязательный комментарий, автора и источник.

Правила:

- минимум 1 минута;
- максимум 24 часа;
- ручной ввод — любое целое число минут;
- элементы `+/-`, если используются, шаг 5 минут;
- будущая дата запрещена;
- в закрытую задачу новое время не добавляется.

Пользователь обычно пишет время только за себя. Отдельные права нужны для другого пользователя, просмотра чужого времени, изменения чужих записей и административных изменений закрытого периода.

Свои записи можно редактировать и мягко удалять, пока период открыт. Закрытие периода блокирует создание/изменение/удаление до заданной даты; исключение требует отдельного права и аудита.

Прогресс не изменяется от времени. Фактическое время — сумма записей; estimate, spent и превышение показываются отдельно.

Нужны экран «Моё время», базовый отчёт и CSV-экспорт по праву. Финансы, billing, payroll, budgets, approval и weekly grid не входят.

## 20. Наблюдение и уведомления

Добровольный наблюдатель не получает дополнительных прав.

Получатели каждого события вычисляются из конкретных пользователей:

- добровольные наблюдатели;
- автор;
- текущий исполнитель;
- новый/предыдущий исполнитель для назначения;
- упомянутые пользователи;
- явно определённые адресаты события.

Дубликаты удаляются, инициатор обычно не уведомляется о своём действии, перед отправкой проверяется актуальный доступ и настройки каналов. Автор и исполнитель не обязаны храниться как добровольные наблюдатели. Добровольная подписка прежнего исполнителя сохраняется.

Каналы 1.0:

- центр уведомлений внутри системы;
- unread count;
- email;
- пользовательские настройки.

Для каждого типа события явно определяются обязательность внутреннего уведомления, возможность отключения, email default и способ отправки. Абстрактной отдельной модели `critical/normal/info` нет.

Обязательные события включают назначения, упоминания, комментарии, статус, возврат, срок, просрочку, удаление/восстановление, потерю доступа и блокирующие связи.

Несколько изменений одного сохранения группируются, но важные события не теряются.

Напоминание отправляется по часовому поясу получателя: один раз утром в день срока и один раз утром следующего дня при просрочке. Время — системная настройка, рекомендуемый default 09:00. Ежедневных повторов нет.

Push, Telegram, SMS, Slack/Teams, дайджесты и email replies не входят.

## 21. Списки, «Моя работа» и доска

Основные рабочие экраны:

1. «Моя работа»;
2. список проекта;
3. доска проекта;
4. глобальный список доступных задач;
5. отдельно Recent.

«Моя работа» содержит:

- требуют внимания;
- назначены мне;
- созданные мной;
- наблюдаемые.

Причина «требует внимания» всегда объясняется: просрочка, срок сегодня, непросмотренное назначение/возврат/упоминание, блокировка. Скрытого AI-рейтинга нет.

Desktop-список — таблица; mobile — карточки. Базовые колонки: ключ, название, проект, тип, статус, исполнитель, приоритет, прогресс, срок, обновление.

Пользователь настраивает колонки, порядок, ширину, закрепление, плотность, сортировку и одну группировку. Состояние хранится в представлении.

Фильтр: набор условий `поле + оператор + значение`; между условиями `И`, несколько значений внутри условия — `ИЛИ`. Вложенных логических групп нет.

Поддерживаются быстрые фильтры: открытые, мои, без исполнителя, просроченные, срок сегодня, недавно обновлённые, с unread.

Серверная пагинация, например 25/50/100. Бесконечная прокрутка не основная модель. При возврате из карточки сохраняются страница, scroll, фильтры и представление.

Массовые операции: исполнитель, приоритет, срок, прогресс, теги, workflow, перенос, тип, soft delete. Каждая задача проверяется отдельно; частичный успех показывает результат по каждой. Выбор всех результатов фильтра требует подтверждения.

Доска строится по статусам workflow. Drag-and-drop выполняет реальный переход и при необходимости открывает форму комментария/полей/подтверждения. Есть альтернативное меню. Объединённая доска проектов возможна только при реально совместимых статусах и порядке.

CSV-экспорт текущего представления — по отдельному праву и с аудитом.

## 22. Сохранённые представления, поиск и навигация

Сохранённое представление включает область данных, фильтры, сортировку, одну группировку, колонки, плотность, подпроекты и режим list/board.

Представления:

- личные — любой пользователь;
- проектные — по праву.

Локальные изменения общего представления не перезаписывают его молча: сохранить, скопировать себе или сбросить.

Глобальный поиск доступен везде, ищет только задачи и проекты и строго соблюдает доступ. Недоступные объекты нельзя раскрывать даже косвенно.

Задачи: ключ, название, описание, комментарии, поддерживаемые текстовые поля. Проекты: название, код, описание. Exact task key имеет высший приоритет. Закрытые задачи ищутся, удалённые — только администратором с фильтром. Архивные проекты видны в поиске только системным администраторам согласно позднему правилу архивирования.

Есть compact results и отдельная страница с фильтрацией, сортировкой, пагинацией и snippets.

Recent — личная очищаемая история открытия задач и проектов, не аудит.

Favorites: проекты и доступные сохранённые представления. Избранных задач нет; для задач используется наблюдение.

Основная навигация: My Work, Projects, All Tasks, Boards, Recent, Notifications, Favorites; Administration отдельно.

## 23. REST API

Обязателен versioned JSON REST API под `/api/v1` или эквивалентным стабильным префиксом. OpenAPI обязателен и должен соответствовать реализации.

API покрывает проекты, задачи, комментарии, вложения, наблюдение, связи, чек-листы, время, workflow-действия, form metadata, справочники, фильтры, сортировку и пагинацию. Полное административное API не требуется.

Обычные пользователи не создают personal API tokens. Интеграции используют service users с обычными ролями и scoped tokens. Эффективный доступ — пересечение прав пользователя и scopes.

Токены:

- всегда имеют срок;
- показываются полностью один раз;
- хранятся безопасно;
- имеют имя, owner, created/expiry/last use, scopes, status, revoked;
- немедленно отзываются;
- инвалидируются при деактивации service user;
- жизненный цикл аудитируется.

API обязан иметь rate limit, request ID, allowlisted filters/sorts, server pagination и стабильный error envelope. Семантика HTTP: 400/401/403/404/409/422/429.

Идемпотентность обязательна минимум для создания задачи, комментария и записи времени. Повтор того же ключа и payload не создаёт дубль; тот же ключ с другим payload возвращает `409`. Срок хранения и внутренняя реализация — решение исполнителя.

Bulk API допускает частичный успех с результатом по объектам и не обходит workflow.

API metadata формы возвращает доступные типы, поля, обязательность, значения, ограничения, порядок/секции и editability. Универсальный schema-builder для всех сущностей не нужен.

Webhooks, OAuth provider, GraphQL, marketplace, Telegram bot, email-to-task, AI и универсальный CSV import не входят.

## 24. Конкурентное редактирование и транзакции

Optimistic locking или эквивалент обязателен для:

- задач;
- проектов и важных настроек;
- workflow;
- типов задач и конфигурации форм;
- других сущностей, где потеря параллельных изменений существенна.

Не требуется механически для каждой простой дочерней записи.

При конфликте пользователь видит сообщение и не затирает молча новые данные.

Атомарны как минимум:

- переход плюс обязательный комментарий/поля;
- смена типа плюс валидация;
- перенос проекта/задачи;
- создание задачи с зависимыми данными;
- удаление/восстановление;
- массовая операция в пределах правил её частичного результата.

## 25. Корзина и физическое удаление

Мягкое удаление применяется к задачам, комментариям, вложениям задач и времени.

Общая административная корзина показывает:

- удалённые задачи;
- отдельно удалённые вложения задач;
- удалённые записи времени.

Комментарии восстанавливаются в задаче или её административном просмотре и не имеют отдельного общего списка.

Удалённые объекты хранятся до ручного физического удаления. Автоочистки корзины в 1.0 нет.

Физическое удаление выполняет только системный администратор после подтверждения и с аудитом. Непустой проект не удаляется физически. Комментарий физически удаляется только вместе с задачей.

## 26. Администрирование и эксплуатация

Административные разделы:

- основные настройки и branding;
- пользователи и безопасность;
- роли, группы и права;
- типы, workflow, поля и справочники;
- почта и уведомления;
- файлы;
- API tokens/service users;
- история, аудит и корзина;
- диагностика.

Логотип можно загрузить, заменить и удалить.

SMTP через UI может включать host, port, encryption, username/password, sender name/address и тестовое письмо. Production-секреты могут переопределяться env.

Почта и тяжёлые действия используют очереди, когда уместно. Временная недоступность email не откатывает основное пользовательское действие.

Базовая диагностика очереди/планировщика:

- доступность очереди;
- failed jobs count;
- последняя ошибка;
- последний успешный scheduler run;
- retry failed job.

Управление worker-процессами из приложения не требуется.

Health checks: приложение, БД, storage, queue и свежесть scheduler; публичный ответ не раскрывает инфраструктуру.

Backup/restore документируются и реально проверяются, но не управляются из UI. Обновления — deployment-процессом.

## 27. Локализация, время и PWA

Обязательные локали интерфейса: **русская и украинская**. Английская локаль не обязательна.

Пользователь выбирает локаль и часовой пояс. Строки не hardcoded, даты и pluralization корректны, API error codes не зависят от перевода. Email/уведомления используют локаль получателя.

Date-only значения не смещаются между часовыми поясами; timestamps показываются в timezone пользователя.

Basic PWA входит в 1.0:

- installable;
- manifest и icon;
- standalone mode;
- правильный viewport;
- безопасное обновление frontend assets.

Offline editing, background sync, conflict resolution и push не входят.

## 28. UI/UX и поддерживаемые клиенты

Обязательные клиенты:

- текущие Chrome и Edge;
- текущий Safari на iPhone;
- текущий Chrome на Android.

Firefox — best effort: Firefox-only дефект не блокирует релиз, если исправление требует отдельной архитектуры, существенных компромиссов или несоразмерного объёма.

Контрольные размеры:

- 1440×900;
- 1024×768;
- 390×844;
- желательно 360×800.

Mobile должен позволять создавать и редактировать задачи, делать переходы, комментировать, загружать файлы, работать с checklist, временем, поиском, фильтрами и уведомлениями. Административные редакторы могут быть удобнее на desktop, но не должны ломаться на mobile.

Обязательны понятные loading/error/empty states, подтверждения, длинные тексты без разрушения layout, отсутствие лишнего horizontal scroll, touch targets, keyboard operation, visible focus, labels, contrast и корректные dialogs.

## 29. Производительность и масштаб

Архитектурный baseline:

- 1 организация;
- до 1 000 активных пользователей;
- до 1 000 проектов/подпроектов;
- до 1 000 000 задач;
- несколько миллионов комментариев, событий и записей времени.

Это архитектурный ориентир, а не обязательная сертификация нагрузочным тестом на миллион записей. Схема, индексы, пагинация и запросы не должны очевидно предполагать сотни записей.

Ориентиры нормального production-окружения:

- обычный API GET — около 500 мс server time;
- сложные списки/отчёты — до 2 секунд;
- обычная карточка — до 2 секунд;
- обычное сохранение — до 1 секунды без длительной фоновой работы.

Это практические цели, а не формальная гарантия без согласованного hardware/load profile. N+1, загрузка полной истории и клиентская пагинация больших наборов недопустимы.

## 30. Безопасность

Обязательны:

- CSRF и XSS protection;
- безопасный Markdown;
- parameterized DB access;
- mass-assignment protection;
- server-side authorization каждой операции;
- login throttling;
- secure cookies;
- request size limits;
- MIME/extension/signature checks;
- безопасные headers;
- HTTPS-only production;
- отсутствие секретов в логах, истории, аудите и API responses.

Frontend visibility не является авторизацией. Недоступные объекты не раскрываются через поиск, counts или exact IDs.

Critical dependency vulnerabilities блокируют релиз. High обычно блокируют, но допускаются как явное документированное исключение при отсутствии безопасного fix или доказанной неприменимости. Исключение содержит риск, компонент, mitigation, owner и review/upgrade plan.

2FA не входит в 1.0.

## 32. CI, миграции, deployment и backup

Каждый PR минимум запускает:

- backend tests;
- frontend typecheck;
- relevant frontend tests;
- production build;
- lint/format;
- migration verification;
- dependency/security checks.

Smoke E2E может идти на каждый PR, полный набор — перед merge/release.

Миграции работают:

- на чистой БД;
- при upgrade существующей;
- без ручного редактирования таблиц;
- с сохранением данных;
- с учётом больших таблиц;
- с rollback или safe recovery plan для рискованных изменений.

Production deployment воспроизводим и документирует runtime, database, frontend build, web server, queues, scheduler, storage, SMTP, HTTPS, permissions и health checks. Docker допустим, но не обязан быть единственным способом.

До приёмки реально выполняется backup/restore drill: данные и файлы → backup → чистое окружение → restore → проверка задач, комментариев, прав, истории, времени и файлов.

## 34. Definition of Done и приёмка 1.0

Функция готова, когда:

- бизнес-правила реализованы;
- Web/API согласованы;
- права проверены server-side;
