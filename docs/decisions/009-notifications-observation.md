# 009. Уведомления и наблюдение

## Статус

Утверждено.

## Решение

### Каналы MVP

В MVP используются:

- уведомления внутри системы;
- email;
- центр уведомлений;
- счетчик непрочитанных;
- пользовательские настройки уведомлений.

Telegram, push, SMS, Slack/Teams, дайджесты, ответы на email и AI-сводки в MVP не входят.

### Наблюдение

Наблюдение за задачей не дает дополнительных прав доступа.

Пользователь, который видит задачу, может:

- добавить в наблюдатели самого себя;
- прекратить собственное наблюдение.

Обычный пользователь не может добавлять в наблюдатели других пользователей. Такое действие доступно только при наличии отдельного проектного права.

Для отдельной задачи в MVP используется только два состояния:

- наблюдать;
- не наблюдать.

Детальная настройка типов событий выполняется глобально в пользовательских настройках.

### Автор и исполнитель

Автор задачи автоматически становится наблюдателем и остается им, пока пользователь активен. Автор не может отписаться от своей задачи.

Текущий исполнитель автоматически становится наблюдателем при назначении.

При смене исполнителя:

- новый исполнитель становится наблюдателем;
- предыдущий исполнитель автоматически перестает быть наблюдателем;
- новый исполнитель получает уведомление о назначении;
- предыдущий исполнитель получает уведомление о снятии назначения;
- событие фиксируется в истории задачи.

### Получатели уведомлений

В зависимости от события уведомления могут получать:

- автор;
- текущий исполнитель;
- наблюдатели;
- упомянутые пользователи;
- новый или предыдущий исполнитель;
- другие адресаты, прямо указанные правилами события.

Все участники проекта не подписываются на все события автоматически.

### Значимые события

Обязательные события уведомлений:

- назначение задачи;
- снятие назначения;
- добавление наблюдателем;
- упоминание пользователя;
- новый комментарий;
- смена статуса;
- возврат задачи в работу;
- изменение срока;
- наступление срока;
- просрочка;
- мягкое удаление;
- восстановление;
- потеря доступа к проекту;
- добавление блокирующей связи;
- закрытие блокирующей задачи.

Дополнительные события могут включать:

- изменение приоритета;
- изменение описания;
- изменение готовности;
- изменение оценки времени;
- добавление вложения;
- изменение чек-листа;
- добавление подзадачи;
- изменение пользовательских полей.

### Комментарии и упоминания

Новый комментарий уведомляет автора, текущего исполнителя, наблюдателей и упомянутых пользователей, кроме автора самого комментария.

Email о новом комментарии отправляется только пользователям, у которых включены email-оповещения о комментариях.

Упомянуть можно только пользователя, который имеет доступ к задаче. Если доступа нет, пользователь не предлагается для упоминания и не получает уведомление.

### Смена статуса и переходы

Смена статуса уведомляет автора, текущего исполнителя и наблюдателей.

Если переход сопровождается обязательным комментарием, смена статуса и комментарий объединяются в одно уведомление.

### Группировка изменений

Несколько изменений, выполненных одним сохранением, объединяются в одну запись активности и одно уведомление.

Назначение, упоминание, просрочка и другие критичные события не должны теряться внутри общей группировки.

### Уровни уведомлений

События делятся на:

- критичные;
- обычные;
- информационные.

Критичные события отображаются внутри системы и могут отправляться по email. Их внутренний канал нельзя полностью отключить.

Обычные события всегда доступны внутри системы, а email зависит от пользовательской настройки.

Информационные события по умолчанию отображаются только внутри системы.

### Центр уведомлений

Центр уведомлений должен поддерживать:

- непрочитанные уведомления;
- группировку по времени;
- проект;
- номер и название задачи;
- автора события;
- краткое содержание;
- отметку одного или всех уведомлений прочитанными;
- фильтры по основным типам событий;
- переход к задаче и конкретному событию, когда это возможно.

### Напоминания о сроке

Для MVP:

- уведомление утром в день срока;
- одно уведомление утром следующего дня, если задача просрочена и не закрыта;
- ежедневные повторные уведомления не отправляются.

### Email

Email должен содержать только полезный контекст:

- тип события;
- проект и номер задачи;
- название;
- автора действия;
- важные изменения;
- комментарий, если он относится к событию;
- текущего исполнителя;
- срок;
- прямую ссылку на задачу.

Ответ на email для добавления комментария в MVP не поддерживается.

### API

Изменения через API создают те же уведомления, что и действия через веб-интерфейс. Бизнес-логика уведомлений едина для всех клиентов.

В уведомлении может отображаться источник события: пользователь, API-интеграция или системный процесс.

## Ограничения MVP

Не входят в MVP:

- индивидуальные правила уведомлений для отдельных задач;
- сложные правила по фильтрам;
- ежедневные и еженедельные дайджесты;
- Telegram;
- push;
- SMS;
- ответы на email;
- умное определение важности;
- AI-сводки;
- эскалации по цепочке руководителей;
- ежедневные повторные письма о просрочке.
