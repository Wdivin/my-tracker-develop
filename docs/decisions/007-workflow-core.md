# 007. Базовая модель workflow

Статус: утверждено.

## Цель

Зафиксировать продуктовые правила workflow без навязывания исполнителям конкретной схемы базы данных или внутренней архитектуры.

## Основные решения

1. Workflow состоит из явных статусов и переходов между ними.
2. Для каждого workflow существует ровно один начальный статус.
3. У статуса обязателен признак, является ли он закрывающим.
4. Категории статусов не используются. Аналитика и фильтрация строятся по конкретным статусам и признаку закрытия.
5. Пользователь изменяет статус только через доступный переход. Универсального действия «перевести в любой статус» нет, в том числе для администратора.
6. Между одной и той же парой статусов допускается только один переход.
7. Если для разных бизнес-сценариев нужен разный смысл результата, используются разные конечные статусы, а не несколько переходов в один и тот же статус.
8. Переход может иметь собственное понятное пользователю название действия, отличное от названия конечного статуса.
9. В карточке задачи пользователь видит только доступные ему действия-переходы, а не произвольный список всех статусов.

## Настройки перехода

Для каждого перехода независимо настраиваются:

- кому он доступен: проектные роли и отношения пользователя к задаче, например исполнитель или автор;
- обязательность комментария;
- набор полей, которые должны быть заполнены перед переходом;
- дополнительные проверки, предусмотренные продуктом;
- уведомления и системные действия, связанные с переходом.

Обязательность комментария задаётся именно для конкретного перехода. Например, переход «Проверка → В работе» может требовать комментарий с причиной возврата, а переход «Новая → В работе» — нет.

## Закрытие задач

1. Если у задачи есть хотя бы одна незакрытая подзадача, перевести родительскую задачу в любой закрывающий статус невозможно.
2. Это жёсткое ограничение целостности процесса, а не предупреждение и не настройка, которую можно обойти правом администратора.
3. При переходе в закрывающий статус применяются остальные правила жизненного цикла задачи: проверка обязательных полей, работа с датой завершения и предложение установить готовность 100%, если это применимо.

## Блокирующие связи

Связь «блокирует / заблокирована» не запрещает переход в закрывающий статус.

Если закрывается задача, которая всё ещё заблокирована незакрытой связанной задачей, система показывает заметное предупреждение и просит подтвердить действие. После подтверждения переход разрешается и фиксируется в истории.

Причина: связь между задачами может устареть или использоваться как информационная. Жёсткий запрет оставляется только для собственных незакрытых подзадач.

## Шаблоны и назначение

1. Workflow назначается типу задачи.
2. Проект может использовать общий шаблон workflow без изменений либо создать независимую копию.
3. Частичные локальные переопределения общего workflow не поддерживаются.

## Интерфейс администратора

Основным представлением редактора должен быть понятный список статусов и переходов с отдельной карточкой настройки перехода.

Дополнительно может использоваться визуальная схема для обзора процесса, но она не должна быть единственным способом настройки.

Перед сохранением система должна проверять целостность workflow, включая как минимум:

- наличие ровно одного начального статуса;
- отсутствие недостижимых статусов;
- отсутствие дублирующих переходов между одной парой статусов;
- корректность ссылок на роли и условия;
- отсутствие очевидно тупиковых состояний, если они не являются закрывающими.

## Не входит в MVP

- переход в произвольный статус;
- несколько переходов между одной и той же парой статусов;
- категории статусов;
- скрипты и произвольные формулы;
- автоматические переходы по таймеру, webhook или входящему письму;
- параллельные согласования и BPMN;
- автоматическое назначение пользователей и скрытое изменение бизнес-полей;
- workflow для отдельных экземпляров задач.
