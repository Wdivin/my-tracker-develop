# Решение 002. Организация, проекты и подпроекты

Статус: утверждено.

## Организация

- На одну установку системы приходится одна организация.
- Организация существует как отдельная доменная сущность.
- Переключение между организациями в MVP отсутствует.
- Организация хранит общие настройки: название, логотип, основной язык, часовой пояс, форматы даты и времени, настройки уведомлений, лимиты вложений и базовые параметры безопасности.
- Архитектура не должна намеренно блокировать возможное добавление multi-tenancy в будущем, однако multi-tenancy не входит в MVP.

## Проект и подпроект

- Проект является основным рабочим контейнером.
- Подпроект не является отдельным типом сущности: это проект с родительским проектом.
- Максимальная глубина дерева проектов ограничена тремя уровнями.
- Циклическая вложенность запрещена.
- Проект можно переносить под другого родителя только при сохранении допустимой глубины и отсутствии циклов.

## Основные свойства проекта

Проект должен иметь как минимум:

- название;
- уникальный код;
- slug для пользовательского URL;
- стабильный публичный идентификатор UUID или ULID;
- описание;
- родительский проект;
- владельца проекта;
- руководителя проекта;
- дату начала;
- плановую дату завершения;
- фактическую дату завершения;
- статус;
- видимость;
- цвет или иконку;
- порядок отображения;
- автора и дату создания.

Код проекта используется в человекочитаемом номере задачи, например `WEB-142`. Изменять код после начала работы может только администратор, с явным предупреждением о влиянии на привычные ссылки и интеграции. Внешние интеграции должны опираться на стабильный UUID или ULID, а не только на изменяемый код.

## Владелец и руководитель проекта

- Владелец проекта отвечает за существование и бизнес-назначение проекта.
- Руководитель проекта отвечает за повседневное управление работой.
- Владелец и руководитель могут быть одним пользователем, но являются разными ролями проекта.
- Руководитель проекта не получает глобальные административные права системы.

## Статусы проекта

Поддерживаются статусы:

- планируется;
- активный;
- приостановлен;
- завершён;
- архивный.

Завершённый проект остаётся доступным для просмотра. Архивный проект скрывается из обычной рабочей навигации и открывается через отдельный раздел. В архивном проекте по умолчанию запрещено создавать и менять задачи, добавлять комментарии, вложения и записи времени.

## Видимость проекта

В MVP поддерживаются два режима:

- закрытый — проект видят только участники и пользователи с глобальным административным доступом;
- внутренний — проект видят все активные пользователи организации, но изменять данные могут только участники с соответствующими правами.

Публичный доступ без авторизации в MVP не входит.

## Конфигурация проекта и наследование

Общесистемными являются:

- типы задач;
- статусы задач;
- роли;
- пользовательские поля;
- шаблоны workflow.

Проект выбирает:

- доступные типы задач;
- workflow для каждого типа;
- активные пользовательские поля;
- доступные проектные роли;
- стандартные сохранённые представления.

Подпроект по умолчанию наследует конфигурацию родителя.

Подпроект может перейти в режим `Использовать собственную конфигурацию`. После этого он получает полностью самостоятельный набор проектных настроек. Частичное смешанное наследование не поддерживается: проект либо наследует конфигурацию целиком, либо использует собственную.

## Участники и наследование участников

- Участник проекта является связью пользователя с проектом и одной или несколькими ролями.
- Один пользователь может иметь несколько ролей в одном проекте.
- Отключение пользователя или удаление его из участников не удаляет его задачи, комментарии, записи времени и историю действий.
- Если задача остаётся назначенной пользователю, потерявшему доступ, система должна показывать предупреждение и позволять переназначить задачу.

Подпроект по умолчанию наследует участников родительского проекта.

Допускаются два режима:

1. наследовать участников родителя и добавлять локальных участников;
2. отключить наследование и использовать собственный список участников.

В интерфейсе обязательно показываются:

- унаследованные участники;
- локальные участники;
- источник унаследованной роли.

Унаследованного участника нельзя удалить непосредственно в подпроекте, пока включено наследование.

## Полномочия руководителя проекта

Руководитель проекта может в пределах своих прав:

- редактировать проект;
- управлять локальными участниками;
- создавать подпроекты;
- завершать и архивировать проект;
- выбирать доступные типы задач;
- назначать workflow;
- настраивать проектные представления;
- управлять тегами проекта;
- просматривать аудит проекта.

Руководитель проекта не может автоматически:

- управлять всеми пользователями организации;
- менять глобальные статусы и типы задач;
- менять системные роли;
- управлять глобальными настройками;
- видеть закрытые проекты, в которых у него нет доступа.

## Агрегирование по дереву

Родительский проект должен позволять просматривать:

- только свои задачи;
- свои задачи и задачи всех подпроектов;
- задачи выбранных подпроектов.

Список задач может агрегироваться всегда.

Объединённая доска по статусам доступна только тогда, когда выбранные проекты используют совместимые workflow и наборы статусов. При несовместимых процессах система должна либо предложить другой способ группировки, либо явно сообщить, почему общая доска недоступна.

## Архивирование дерева

Нельзя архивировать проект, пока у него существуют активные подпроекты.

Пользователь должен сначала:

- завершить или архивировать активные подпроекты;
- либо перенести их в другую ветвь дерева.

Система должна показать понятное сообщение и список подпроектов, мешающих архивированию.

## Удаление проекта

Обычный рабочий сценарий — завершение и архивирование, а не физическое удаление.

Физически удалить проект может только администратор и только если проект полностью пуст:

- нет задач;
- нет комментариев;
- нет вложений;
- нет записей времени;
- нет подпроектов;
- нет других связанных рабочих данных.

Непустой проект можно только архивировать.

## Избранные и недавно открытые проекты

Каждый пользователь может добавлять проекты в избранное.

В навигации должны быть отдельные удобные блоки:

- избранные проекты;
- недавно открытые проекты;
- все доступные проекты.

Избранность является персональной настройкой пользователя и не влияет на права или видимость проекта.

## Мобильная навигация

На мобильных устройствах не требуется показывать большое рекурсивное дерево в боковой панели.

Предпочтительный пользовательский сценарий:

- хлебные крошки или компактное отображение пути;
- быстрый переход к родительскому проекту;
- отдельный список дочерних проектов;
- поиск проекта;
- избранные проекты;
- недавно открытые проекты.

Конкретное визуальное решение выбирает исполнитель, но оно должно быть проверено через браузерный MCP на мобильных разрешениях.
