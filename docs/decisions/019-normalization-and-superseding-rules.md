# Решение 019. Нормализация утверждённых требований и приоритет поздних решений

## Статус

Утверждено.

## Назначение

Этот документ устраняет противоречия и неоднозначности между ранней спецификацией продукта и более поздними решениями.

При конфликте требований действует следующий приоритет:

1. мастер-бриф версии 1.0;
2. более позднее утверждённое решение из `docs/decisions`;
3. более раннее решение из `docs/decisions`;
4. `docs/PRODUCT_SPEC.md`;
5. README и прочие пояснительные документы.

Позднее решение не требует физического удаления раннего текста, но явно заменяет противоречащую ему формулировку.

## Терминология релизов

Полный утверждённый объём продукта называется **целевой версией 1.0**.

Реализация должна быть разбита на последовательные этапы. Первый этап — **рабочее ядро**, пригодное для реального использования. Последующие этапы доводят продукт до полного объёма версии 1.0.

Требование версии 1.0 не означает, что весь объём должен реализовываться одним большим изменением.

## Active Directory и LDAP

Интеграция с Active Directory/LDAP исключается из версии 1.0 и переносится на последующее развитие.

Для версии 1.0 остаются:

- локальные пользователи;
- создание пользователей системным администратором;
- приглашение и установка пароля;
- восстановление пароля;
- деактивация без удаления истории;
- локальная аварийная учётная запись системного администратора.

Требования Decision 017 к доменной интеграции считаются отложенными и будут уточняться перед отдельной реализацией.

## Системные администраторы

Системный администратор может:

- назначать другим пользователям права системного администратора;
- снимать права системного администратора у других пользователей;
- управлять глобальными настройками, пользователями, правами и восстановлением данных.

Это уточнение не создаёт универсального обхода workflow обычных задач и не разрешает редактировать или подменять аудит.

## Проекты и подпроекты

### Глубина дерева

Разрешено не более двух уровней подпроектов:

```text
Корневой проект
└── Подпроект первого уровня
    └── Подпроект второго уровня
```

Более глубокая вложенность запрещена.

### Код проекта

- Код задаётся при создании проекта.
- После создания проекта код изменить нельзя.
- Код используется в человекочитаемых ключах задач.
- UUID или ULID используется как стабильный технический идентификатор и для интеграций.

### Slug

- Slug автоматически формируется из названия проекта.
- Он приводится к безопасному для URL виду и делается уникальным.
- Пользователь с соответствующим правом может изменить slug.
- Изменение slug не меняет UUID/ULID и не должно ломать внутренние связи.
- Поддержка полной истории старых slug и сложной системы редиректов не обязательна для версии 1.0.

### Статусы проекта

- `планируется` — работа разрешена по правам;
- `активный` — обычный рабочий режим;
- `приостановлен` — проект виден участникам, рабочие изменения по умолчанию запрещены;
- `завершён` — проект доступен участникам для просмотра, новые задачи и обычные изменения запрещены;
- `архивный` — проект скрыт для всех, кроме системных администраторов.

Возврат завершённого или архивного проекта в рабочее состояние требует отдельного права, подтверждения и аудита.

### Наследование

Наследование конфигурации и наследование участников являются независимыми режимами.

Подпроект может:

- наследовать конфигурацию родителя целиком;
- либо использовать собственную конфигурацию.

В конфигурацию входят:

- доступные типы задач;
- workflow для каждого типа;
- доступные проектные роли;
- проектные представления;
- глобальный или собственный полный набор видов работ.

Пользовательские поля проект напрямую не включает. Набор полей определяется типом задачи.

Подпроект отдельно может:

- наследовать участников и роли родителя с возможностью добавлять локальные;
- либо отключить наследование и использовать собственный состав.

## Группы

Группы используются только для управления правами.

Группа:

- может быть добавлена в проект и получить проектную роль;
- не может быть исполнителем задачи;
- не может быть автором;
- не может быть наблюдателем;
- не является получателем уведомлений;
- не может быть владельцем записи времени.

Уведомления всегда адресуются конкретным пользователям.

## Задачи и inline-редактирование

У задачи может быть максимум один исполнитель.

Inline допускается для обычных рабочих полей, в том числе:

- название;
- описание;
- исполнитель;
- приоритет;
- дата начала;
- срок;
- прогресс;
- оценка времени;
- теги;
- простые пользовательские поля;
- наблюдение за задачей;
- элементы чек-листа в собственном блоке.

Через отдельное действие, workflow-переход или мастер выполняются:

- смена статуса;
- перенос в другой проект;
- смена типа задачи;
- изменение родительской задачи;
- удаление и восстановление;
- структурные операции с существенными последствиями.

Локальное автосохранение незавершённой формы создания не входит в первый рабочий релиз.

Нельзя мягко удалить задачу, пока у неё есть неудалённые подзадачи. Сначала их нужно удалить, перенести или отвязать от родителя.

## Типы задач и workflow

Глобально существуют типы задач и шаблоны workflow.

Для каждой пары `проект + тип задачи` проект выбирает:

- глобальный шаблон workflow;
- либо независимую локальную копию шаблона.

Начальный статус определяется только workflow. Тип задачи не хранит отдельный начальный статус.

Отдельных черновиков workflow нет. Сохраняется только целостная конфигурация.

Незакрывающий статус без исходящих переходов вызывает предупреждение, но не безусловную ошибку. Администратор может подтвердить такую конфигурацию. Закрытость определяется только явным признаком статуса, а не его названием.

При закрытии задачи система предлагает установить прогресс 100%, но пользователь может отказаться. Закрытая задача может иметь прогресс меньше 100%. Конкретный переход может требовать определённый прогресс как явное условие.

## Комментарии workflow-перехода

Обязательный комментарий перехода:

- создаётся как обычный комментарий пользователя;
- связан с событием перехода;
- отображается в общей ленте;
- может редактироваться автором по обычным правилам;
- после редактирования помечается как изменённый;
- сохраняет историю редакций.

Редактирование комментария не отменяет и не пересматривает выполненный workflow-переход.

## Наблюдение и уведомления

Получатели вычисляются по событию из конкретных пользователей:

- добровольные наблюдатели;
- автор задачи;
- текущий исполнитель;
- новый и предыдущий исполнитель для событий назначения и снятия;
- упомянутые пользователи;
- иные прямо определённые получатели конкретного события.

Дубликаты получателей удаляются. Инициатор обычно не получает уведомление о собственном действии. Перед отправкой проверяется текущий доступ к задаче и применяются пользовательские настройки каналов.

Автор и исполнитель не обязаны физически храниться как добровольные наблюдатели.

Если прежний исполнитель самостоятельно наблюдал за задачей, его добровольная подписка сохраняется после снятия назначения.

Абстрактные уровни `критичное / обычное / информационное` не используются как отдельная настраиваемая модель. Для каждого типа события явно определяется:

- создаётся ли внутреннее уведомление;
- может ли пользователь его отключить;
- отправляется ли email по умолчанию;
- может ли пользователь отключить email;
- требуется ли немедленная отправка или допустима очередь.

Напоминания о сроке рассчитываются по часовому поясу получателя. Время отправки задаётся системной настройкой, рекомендуемое значение по умолчанию — 09:00.

## Учёт времени

- Минимальная длительность записи — 1 минута.
- Максимальная длительность одной записи — 24 часа.
- Ручной ввод допускает любое целое количество минут в этом диапазоне.
- Элементы управления `+/-`, если используются, меняют значение шагом 5 минут.
- Запись времени в будущем запрещена.
- В закрытую задачу новое время не добавляется.

## REST API и инфраструктурные требования

### Идемпотентность

Идемпотентность обязательна как минимум для:

- создания задачи;
- создания комментария;
- создания записи времени.

Клиент передаёт `Idempotency-Key`. Повтор идентичного запроса с тем же ключом не создаёт дубль. Повтор с тем же ключом и другим содержимым возвращает конфликт `409`.

Срок хранения ключей и внутреннюю реализацию определяет исполнитель.

### Optimistic locking

Обязателен для:

- задач;
- проектов и важных проектных настроек;
- workflow;
- типов задач и конфигурации форм;
- иных конфигурационных сущностей, где потеря параллельных изменений существенна.

Не требуется универсально для каждой простой дочерней записи.

### Метаданные форм

API отдаёт практические метаданные форм задач:

- доступные типы задач;
- поля выбранного типа;
- обязательность;
- доступные значения;
- простые ограничения;
- порядок и секции;
- возможность редактирования.

Универсальный конструктор схем для любых будущих сущностей не требуется.

### Корзина

В общей административной корзине показываются:

- удалённые задачи;
- отдельно удалённые вложения задач;
- удалённые записи времени.

Комментарии восстанавливаются внутри задачи или административного просмотра конкретной задачи и не выводятся отдельным общим списком корзины.

Удалённые объекты хранятся до ручного физического удаления. Автоматической очистки корзины в версии 1.0 нет.

### SMTP и фоновые процессы

Через интерфейс могут настраиваться host, port, encryption, username, password, имя и адрес отправителя, а также тестовая отправка. Допускается переопределение production-секретов переменными окружения.

Базовая диагностика очереди и планировщика включает:

- доступность очереди;
- количество failed jobs;
- последнюю ошибку;
- время последнего успешного запуска scheduler;
- возможность повторить failed job.

Полноценная панель управления worker-процессами не требуется.

## Корзина и физическое удаление

- Автор может восстановить свой мягко удалённый комментарий, пока задача доступна.
- Пользователь с отдельным правом может восстановить чужой комментарий.
- Системный администратор может восстановить комментарий через административный просмотр.
- Комментарии отдельно физически не удаляются.
- Физическое удаление комментариев выполняется только вместе с окончательным удалением задачи.
- Физическое удаление выполняет только системный администратор после явного подтверждения.
- Операция фиксируется в системном аудите.

## Файлы и вложения

### Квоты и значения по умолчанию

В квоту задачи входят только активные вложения.

Стартовые значения:

- 25 МБ на один файл;
- 10 файлов за одну операцию загрузки;
- 500 МБ активных вложений на одну задачу.

Значения настраиваются системным администратором.

Мягко удалённые файлы физически занимают хранилище до ручной очистки, но не учитываются в пользовательской квоте задачи.

### Вложения редакций комментариев

Файлы старых редакций комментария не хранятся.

История редакции может показать, что в прежней версии присутствовал файл или изображение, и сохранить его метаданные, но сам файл старой редакции недоступен.

При замене файла старый файл исключается из текущего комментария, а новый становится текущим вложением. Полноценное версионирование файлов не создаётся.

### Правила типов файлов

Приоритет правил:

1. системно опасные форматы блокируются всегда;
2. явный denylist имеет приоритет;
3. если allowlist заполнен, разрешены только перечисленные типы;
4. если allowlist пуст, разрешено всё, кроме системно опасных и явно запрещённых типов;
5. проверяются расширение, MIME и сигнатура, когда это технически возможно.

ZIP:

- хранится как обычное вложение;
- доступен только для скачивания;
- не распаковывается;
- не исполняется;
- не просматривается inline;
- его содержимое не индексируется.

### Незавершённые загрузки

- не считаются вложениями;
- не входят в квоту;
- временные файлы имеют ограниченный срок жизни;
- просроченные временные загрузки очищаются автоматически;
- повторное завершение не должно создавать дубль.

Контроль целостности хранилища, поиск orphan-файлов и отдельная диагностическая CLI-команда не входят в версию 1.0, но остаются в плане дальнейшего развития.

## История задачи

- История хранится весь срок существования задачи.
- После физического удаления задачи её пользовательская история также удаляется.
- История загружается порциями и фильтруется на сервере.
- Старые события не загружаются полностью при открытии карточки.
- Для коротких полей сохраняются старое и новое значения.
- Для длинного текста используется история версий или отдельное сравнение.
- Для коллекций сохраняются добавленные и удалённые элементы.
- Для файлов сохраняются метаданные и факт операции, но не содержимое.
- Секреты никогда не сохраняются в старых и новых значениях.

## Системный аудит

- Значение хранения по умолчанию — 365 дней.
- Автоматическую очистку можно отключить.
- В версии 1.0 используется единый срок хранения для всего системного аудита.
- Записи нельзя редактировать или выборочно удалять через интерфейс.
- Очистка выполняется только по общей retention-политике.
- После очистки создаётся агрегированная запись с периодом, количеством удалённых записей, инициатором и результатом.
- Криптографический или WORM-аудит не требуется.
- IP-адрес и user agent сохраняются, когда применимы.
- Для системных процессов они могут отсутствовать.
- Длина user agent ограничивается.
- IP и user agent доступны только администраторам и удаляются вместе с аудитом.
- Пароли, полные токены, SMTP-секреты и другие чувствительные значения в аудит не попадают.

Просмотр содержимого и редакций мягко удалённых комментариев регулируется отдельным проектным правом. По умолчанию оно включается в шаблон роли руководителя проекта. Системный администратор имеет глобальный доступ.

## Нефункциональные уточнения

### Firefox

Firefox поддерживается в режиме best effort. Проблема, возникающая только в Firefox, не блокирует выпуск, если её исправление требует отдельной архитектуры, существенных компромиссов или несоразмерного объёма работ.

### Производительность

Утверждённые значения времени ответа являются практическими архитектурными ориентирами, а не формальной сертификацией без заранее заданного оборудования, профиля данных и нагрузки.

### Масштаб

Миллион задач является архитектурным ориентиром. Схема данных, индексы, пагинация и запросы не должны иметь очевидного ограничения на несколько сотен записей. Полная нагрузочная сертификация на миллион задач не входит в версию 1.0.

### PWA

Базовая устанавливаемая PWA остаётся частью версии 1.0. Offline-редактирование, background sync и push-уведомления не входят.

## Упрощения и отложенные возможности

После версии 1.0 переносятся:

- Active Directory/LDAP;
- переключение файлового backend через UI;
- миграция файлов между local и S3 из интерфейса;
- универсальная корзина всех сущностей;
- сложная API-аналитика;
- управление инфраструктурой очереди из приложения;
- расширенные health checks внешних сервисов;
- отдельные retention-политики для каждой категории логов;
- универсальная платформа интеграций;
- webhooks;
- универсальный CSV-импорт;
- полное административное API;
- контроль целостности файлового хранилища и поиск orphan-файлов.
