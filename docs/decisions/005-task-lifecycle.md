# Решение 005. Жизненный цикл задачи

Статус: утверждено.

## Создание задачи

Задача может создаваться из глобальной кнопки, внутри проекта, из доски, как подзадача и через API.

Минимально обязательны проект, тип задачи и название. Остальные обязательные поля определяются типом задачи и workflow.

Автоматически заполняются автор, номер, дата создания, стартовый статус, готовность 0% при включённом поле готовности, источник создания и версия записи.

Незавершённая форма локально автосохраняется в браузере для защиты от случайной потери данных. Отдельная серверная сущность черновика в MVP не требуется.

## Редактирование

Основные поля редактируются inline, без обязательного отдельного режима редактирования. Каждое изменение проходит проверку прав и workflow, записывается в историю и при необходимости создаёт уведомления.

Для защиты от потери данных при одновременном редактировании используется optimistic locking или эквивалентный надёжный механизм. Молчаливое затирание чужих изменений недопустимо.

## Смена статуса

Статус меняется только через разрешённый workflow-переход с учётом текущего статуса, типа задачи, роли, прав и обязательных полей.

Интерфейс показывает только допустимые следующие статусы. API использует те же правила.

Для перехода может настраиваться режим комментария: не требуется, предлагается или обязателен. При повторном открытии задачи комментарий по умолчанию не обязателен.

## Закрытие

Закрытость определяется свойством статуса, а не его названием.

При закрытии фиксируется дата завершения, проверяются обязательные поля и предлагается установить готовность 100%, если она меньше 100%.

Закрытие родительской задачи запрещено, пока у неё есть незакрытые подзадачи. Обход запрета правами руководителя не предусмотрен.

## Повторное открытие

Повторное открытие выполняется обычным workflow-переходом из закрывающего статуса.

Текущее поле даты завершения очищается, история прошлых закрытий сохраняется. Готовность автоматически не уменьшается. Событие фиксируется отдельно, наблюдатели уведомляются.

## Перенос между проектами

Перенос разрешён только пользователю с необходимыми правами в исходном и целевом проектах.

Перенос выполняется через мастер, который проверяет доступность типа задачи, workflow, текущего статуса, исполнителя и пользовательских полей. Пользователь заранее видит, какие значения будут изменены, скрыты или сброшены.

Молчаливое обнуление несовместимых данных запрещено.

## Изменение типа задачи

Изменение типа разрешено через аналогичный мастер с показом последствий.

Если меняются workflow, обязательные поля или доступный набор полей, система должна явно показать пользователю последствия. Значения скрытых пользовательских полей не удаляются физически и могут восстановиться при обратной смене типа.

## Архивирование

Отдельного состояния «архивная задача» нет.

Задачи бывают открытыми, закрытыми и мягко удалёнными. Архивирование относится к проектам, а не к отдельным задачам.

## Удаление и восстановление

Удаление задачи является мягким: задача скрывается из обычных списков, но сохраняются история, комментарии, вложения, связи и аудит удаления.

Восстановление мягко удалённой задачи доступно только системному администратору.

Физическое удаление допускается только системным администратором из корзины, с явным подтверждением и аудитом. Автоматическая очистка корзины не обязательна для MVP.

## Копирование

Функция копирования или клонирования задачи в MVP не включается.

Связь «дублирует / дублируется» может использоваться как обычный тип связи между задачами без отдельной команды копирования.

## Массовые операции

Массовые изменения статуса, исполнителя, приоритета, срока и тегов проверяются отдельно для каждой задачи. Допустим частичный успех с понятным результатом по каждой записи.

Массовая операция не может обходить права или workflow.

## Техническое состояние

Бизнес-статус задачи отделён от технического состояния.

Бизнес-статус описывает процесс работы. Техническое состояние определяет, активна задача или мягко удалена. Закрытость определяется свойством бизнес-статуса.
