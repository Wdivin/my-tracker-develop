# Принципы продукта My Tracker

Статус: утверждено.

## Главная цель

Создать современный трекер задач, который сохраняет гибкость корпоративных систем уровня Redmine и Jira, но остаётся настолько простым и интуитивным, что новый сотрудник сможет начать работать практически без обучения, одинаково комфортно используя компьютер или телефон.

## Инженерный девиз

**Сложность должна жить внутри системы, а не в голове пользователя.**

## P-001. Мы проектируем продукт, а не копируем существующие системы

Redmine, Jira, YouTrack, Asana, Trello, Linear и другие продукты используются как источник удачных и неудачных решений, но не как обязательный эталон.

## P-002. Простота важнее количества функций

Сложность должна открываться постепенно. Новичок видит необходимый минимум, опытный пользователь получает расширенные возможности.

## P-003. Mobile — полноценное рабочее место

Мобильный интерфейс проектируется как самостоятельный удобный сценарий, а не как уменьшенный desktop.

## P-004. Одно действие — один ответственный

У задачи один основной исполнитель. Совместная работа организуется через подзадачи, наблюдателей, комментарии и связи.

## P-005. Никакой скрытой магии

Наследование, автоматизация и последствия действий должны быть понятны пользователю и администратору.

## P-006. API — равноправный клиент системы

Web UI и API используют одни бизнес-правила, права доступа и workflow.

## P-007. История важнее удаления

Ценные рабочие данные не должны исчезать безвозвратно. По умолчанию применяются архивирование, отключение и мягкое удаление.

## P-008. Безопасные настройки по умолчанию

Система не должна случайно открывать доступ, изменять большое количество данных или запускать необратимые действия.

## P-009. Каждое усложнение должно окупаться

Новая функция оправдана, если она экономит время, повышает качество работы или существенно расширяет возможности продукта.

## P-010. Решения проверяются с четырёх сторон

Перед утверждением существенной идеи проверяется:

1. Пользователю стало проще?
2. Руководителю стало понятнее?
3. Администратору стало легче поддерживать систему?
4. Разработчику не достанется неоправданный технический долг?

Если одна из проверок не проходит, решение требует дополнительного обсуждения.

## P-011. Принятые решения можно улучшать

Документация отражает лучшее текущее понимание продукта. Если найдено более качественное решение, старое решение можно заменить с фиксацией причины в журнале решений и `WHY_NOT.md`.

## Роли в проектировании

- пользователь проекта выступает Product Owner и принимает окончательные продуктовые решения;
- ChatGPT выступает Solution Architect и Product Challenger, предлагает варианты и ищет слабые места;
- Codex и Claude Code независимо реализуют согласованную спецификацию и имеют свободу технической реализации в заданных продуктовых границах.
